Quo Vadis — Push Notification System (v1.21) — Key Notes
=========================================================

ARCHITECTURE:
- Client (app.js) calls queuePushNotification(type, data)
- Writes to Firebase RTDB: notifications/queue/{pushId}
- Cloud Function (sendPushNotification) triggers on onValueCreated
- Cloud Function: collectTokens → filterByTarget → filterChatToggle → FCM → dispatch

TOKEN STRUCTURE:
- fcm_tokens/{uid}/{deviceId} → { token, role, device, uid, name, updatedAt }
- Multi-device: ogni dispositivo ha il proprio token
- Cleanup automatico: token invalidi rimossi dalla Cloud Function dopo errore FCM

ROLE ASSIGNMENT:
- UID in OWNER_UIDS → owner
- Loggato con Google, UID non in OWNER_UIDS → family
- Non loggato (nessun token registrato) → (non riceve push)

CLOUD SCHEDULER:

TEST PUSH BUTTON:

- Funzione: Scrive una notifica di test direttamente nella coda Firebase
- Flusso: Click → push to notifications/queue → Cloud Function trigger → FCM → dispositivo

TECHNICAL NOTES:
1. Countdown solo server-side: gestito dalla Cloud Function dailyCountdown
2. Nessun rate-limiting server-side: dedup è client-side via sessionStorage
3. Chat notifications opt-in: default OFF, l'utente deve cliccare 🔔
4. Sender exclusion: campo senderUid nel payload esclude l'autore
5. Token cleanup: Cloud Function rimuove token con errori messaging/registration-token-not-registered
6. Tag FCM: campo tag nel payload permette al browser di raggruppare/sostituire notifiche dello stesso tipo
7. Zaino threshold: zaino_reminder si attiva solo se ci sono più di 10 oggetti non spuntati
